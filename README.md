# Εντοπισμός Υποχρεώσεων Υποβολής Αναφοράς στην Ελληνική Νομοθεσία

Διπλωματική εργασία : Ανάπτυξη Συνόλου Δεδομένων (dataset) για τον εντοπισμό υποχρεώσεων υποβολής αναφορών (Reporting Obligations, RO) στην ελληνική νομοθεσία.

## Περιγραφή

Το σύστημα επεξεργάζεται νομικά κείμενα από βάση δεδομένων harvester.db και ανιχνεύει προτάσεις που συνιστούν υποχρεώσεις αναφοράς, μέσω ενός rule-based ταξινομητή. Η
ροή είναι:

1. Ανάγνωση νόμων από τη βάση harvester.db.
2. Διάσπαση σε προτάσεις (`legal_splitter_gr.py`).
3. Ανίχνευση υποψήφιων προτάσεων και ταξινόμηση κατά EURO-5K
   (`reporting_obligations_v1.py`, `euro5k_patterns.py`).
4. Παραγωγή JSON.
5. Δημιουργία dataset με 200 τυχαία δείγματα 

Παρέχεται επίσης LLM baseline (`llm_classifier05.py`) για σύγκριση (χρήση μέσω openrouter)


## Το παραγόμενο αρχείο εξόδου (reporting_obligations.json)

Το reporting_obligations.json είναι το τελικό "προϊόν" του pipeline. Περιέχει μια λίστα από εγγραφές — μία για κάθε πρόταση νόμου, που το script ξεχώρισε ως πιθανή υποχρέωση αναφοράς και την αξιολόγησε. Κάθε εγγραφή είναι ένα αντικείμενο JSON με τα εξής πεδία:

id — μοναδικό αναγνωριστικό της εγγραφής (π.χ. GR_OBL_0001), για εύκολη παραπομπή.

text — η ίδια η πρόταση που αξιολογείται. Αυτή είναι το επίκεντρο: το σύστημα κρίνει αυτήν ως υποχρέωση αναφοράς ή όχι.

previous_sentence / next_sentence — η προηγούμενη και η επόμενη πρόταση στο κείμενο του νόμου. Δίνονται ως συμφραζόμενα, ώστε να είναι κατανοητό το πλαίσιο της υπό εξέταση πρότασης.

label — η ετικέτα της ταξινόμησης: 1 αν η πρόταση χαρακτηρίστηκε ως υποχρέωση αναφοράς, 0 αν όχι.

metadata — πληροφορίες για την απόφαση του συστήματος:

trigger_phrase: η φράση που ενεργοποίησε την ανίχνευση (π.χ. «υποχρεούται»).

explanation: σύντομη αιτιολόγηση. Για τις θετικές (label 1) περιγράφει τα στοιχεία που εντοπίστηκαν (υπόχρεος, αποδέκτης, συχνότητα)· για τις αρνητικές (label 0) αναφέρει τον λόγο απόρριψης (π.χ. no_recipient — δεν βρέθηκε αποδέκτης-αρχή).


source — η προέλευση της πρότασης στη νομοθεσία: αριθμός νόμου (law_number), αριθμός ΦΕΚ (fek_number), έτος (year) και άρθρο (article).

Το αρχείο περιέχει τόσο τις θετικές όσο και τις αρνητικές περιπτώσεις, ώστε να μπορεί να χρησιμοποιηθεί ως πλήρες σύνολο δεδομένων (dataset) για εκπαίδευση ή αξιολόγηση μοντέλων ταξινόμησης.

## Δομή

| Αρχείο | Περιγραφή |
|---|---|
| `euro5k_patterns.py` | Λεξικά μοτίβων (deontic, reporting, recipients, exclusions) |
| `reporting_obligations_v1.py` | Κύρια λογική ανίχνευσης και ταξινόμησης |
| `legal_splitter_gr.py` | Διάσπαση νομικού κειμένου σε προτάσεις |
| `llm_classifier_v1.py` | LLM baseline (GPT-4o-mini μέσω OpenRouter) |
| `random200_v1.py` | Δειγματοληψία 100 label=1 + 100 label=0 από το json|

## Εγκατάσταση

```bash
# Παραγωγή json

# LLM baseline (απαιτεί κλειδί OpenRouter ως environment variable)
export OPENROUTER_API_KEY="sk-or-..."   # Windows: set OPENROUTER_API_KEY=...
python llm_classifier_v1.py
```

## Σημείωση για τα credentials

Το `llm_classifier_v1.py` διαβάζει το κλειδί OpenRouter **αποκλειστικά** από τη μεταβλητή περιβάλλοντος `OPENROUTER_API_KEY`. Δεν αποθηκεύεται ποτέ κλειδί
μέσα στον κώδικα ή στο repository. Μπαίνει χειροκίνητα στο τερματικό.
